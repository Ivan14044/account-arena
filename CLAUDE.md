# CLAUDE.md — правила работы над проектом Account Arena

> Этот файл читается Claude Code в начале сессии. Правила ниже **обязательны**.

## 1. Git-процесс: всегда изолированная ветка → PR → merge в `main`

- **Никогда не коммить и не пушь напрямую в `main`.** Любая работа ведётся в отдельной ветке.
- Создавай ветку с понятным префиксом: `feat/…`, `fix/…`, `refactor/…`, `chore/…`, `security/…`.
- Делай маленькие осмысленные коммиты (Conventional Commits: `fix(scope): …`).
- По завершении работы: **открой Pull Request и смержи его в `main`** (squash/merge), затем удали ветку.
- В описании PR перечисляй: что сделано, какие баги закрыты (с ID), и как это проверено.
- Завершай тело PR строкой:
  `🤖 Generated with [Claude Code](https://claude.com/claude-code)`
- Завершай сообщения коммитов строкой:
  `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`

## 2. Приоритет — максимальное качество, а не скорость

- Цель — **самый правильный и полный результат**, а не быстрый. Не торопись, не срезай углы.
- **Проверяй каждое изменение** перед коммитом: запускай тесты, сборку, линтеры, статический анализ; добавляй таргетные регресс-тесты к фиксам.
- Никаких заглушек/`TODO`-обрезков вместо реализации. Доводи до конца.
- **Адверсариально перепроверяй** находки (в т.ч. от субагентов) — не тащи в код непроверенные утверждения. Если находка оказалась ложной — фиксируй это и не «чини» зря.
- Не трогай высокорисковый код (платежи, авторизация, балансы) без верификации. Если верифицировать нечем — сначала подними окружение/тулчейн.
- Behaviour-preserving по умолчанию: рефакторинг не должен менять поведение; изменения поведения — только осознанно и с тестом.

## 3. Окружение и верификация

В системе **нет** глобальных PHP/Node/Composer. Переносимый тулчейн стоит в `~/.local/toolchain`.
Перед проверками добавь его в PATH:
```bash
export PATH="$HOME/.local/toolchain/php:$HOME/.local/toolchain/node/bin:$HOME/.local/toolchain/bin:$PATH"
```
- Backend (Laravel): `cd backend && php artisan test` — тесты на sqlite `:memory:` (см. `phpunit.xml`). Ряд падений — пред-существующие sqlite-артефакты (collation, `/`→404). Также: `php -l`, `php artisan route:list`.
- Frontend (Vue 3/Vite): `cd frontend && npx vite build && npx vue-tsc --noEmit`. eslint содержит много пред-существующего prettier-шума — это не реальные баги.
- `bootstrap/cache/*.php` — генерируемый кэш, не коммить (`git checkout -- backend/bootstrap/cache/`).
- В `.gitignore` стоит `*.md`, поэтому документацию/этот файл добавляй через `git add -f`.

## 4. Архитектура (кратко)

- `backend/` — Laravel: REST API (`routes/api.php`), Blade-админка и кабинет поставщика (`routes/web.php`), SEO/SSR (`Seo\SpaController`), платежи Cryptomus/Monobank, Telegram-бот, KASM-запуск аккаунтов.
- `frontend/` — Vue 3 + Vuetify + Pinia + Vue Router + i18n (Vite).
- Полная карта функционала: `docs/functionality/`. Аудит багов и план ремедиации: `docs/bugs/`, `docs/REMEDIATION.md`.

## 5. Безопасность

- Санитизируй любой серверный/пользовательский HTML перед рендером (фронт: `v-safe-html`/DOMPurify; admin-blade: `escapeHtml`; SSR JSON-LD: `JSON_HEX_TAG`).
- Финансовые операции и вебхуки — идемпотентны (атомарный compare-and-set по статусу транзакции) и под блокировками.
- Проверяй владение (ownership) и роли на каждом мутирующем эндпоинте.
