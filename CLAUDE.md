# CLAUDE.md — правила работы над проектом Account Arena

> Этот файл читается Claude Code в начале сессии. Правила обязательны.
> **Сам этот файл и правила меняем тоже только через ветку → PR → merge.**

## 1. Git-процесс: всегда изолированная ветка → PR → merge в `main`

- **Никогда не коммить и не пушь напрямую в `main`** — даже документацию и правки этого файла. (Ветка `main` технически не защищена — дисциплина на нас.)
- Ветка с понятным префиксом: `feat/ fix/ refactor/ chore/ security/ docs/`.
- Маленькие осмысленные коммиты, Conventional Commits: `fix(scope): …`. Заголовок — по-английски, тело можно по-русски.
- **Стратегия слияния — merge-commit** (`gh pr merge <n> --merge --delete-branch`): сохраняем гранулярную историю по каждому фиксу (важно для трассировки безопасности). `--squash` — только если ветка содержит шумные WIP-коммиты.
- После мерджа ветка удаляется; синхронизируй локальный `main` (`git checkout main && git pull`).
- **Открывать PR и мерджить мне разрешено** (gh аутентифицирован как владелец репо). Порядок: открыл PR → дождался зелёной верификации (см. §2–3) → смержил.
  - **Исключение (важно):** для высокорисковых изменений — платежи, авторизация/доступы, миграции, всё, что трогает прод-данные или внешние интеграции, — сначала покажи краткую сводку диффа и дождись подтверждения, потом мердж.
- Тело PR заканчивай: `🤖 Generated with [Claude Code](https://claude.com/claude-code)`.
- Сообщения коммитов заканчивай: `Co-Authored-By: Claude <noreply@anthropic.com>` (с указанием актуальной модели).

## 2. Приоритет — максимальное качество, а не скорость

- Цель — **самый правильный и полный результат**, а не быстрый. Не торопись, не срезай углы, доводи до конца. Никаких заглушек/`TODO`-обрезков вместо реализации.
- **Верифицируй каждое изменение перед коммитом:** тесты, сборка, линтеры, `php -l`. К каждому фиксу — таргетный регресс-тест.
- **Самопроверка перед мерджем:** перечитай весь дифф свежим взглядом (`git diff main...HEAD`) и/или прогони `/code-review`.
- **Адверсариально перепроверяй находки** (в т.ч. от субагентов) — не тащи непроверенные утверждения в код. Ложные/переоценённые — помечай и НЕ «чини» зря.
- **Качество в рамках согласованного объёма:** не раздувай scope молча. Смежную проблему фиксируй в `docs/REMEDIATION.md` и спроси, а не «чини всё подряд».
- Не трогай высокорисковый код (платежи, авторизация, балансы) без верификации. Нечем верифицировать — сперва подними окружение/тулчейн (§3).
- **Behaviour-preserving по умолчанию:** рефакторинг не меняет поведение; изменение поведения — только осознанно и с тестом.

## 3. Окружение и верификация

В системе **нет** глобальных PHP/Node/Composer — переносимый тулчейн в `~/.local/toolchain`:
```bash
source ~/.local/toolchain/env.sh   # PHP 8.3 + Node 22 + Composer → в PATH
```
- **Backend (Laravel):** `cd backend && php artisan test` — тесты на sqlite `:memory:` (см. `phpunit.xml`).
  **Критерий приёмки:** не вводить **НОВЫХ** падений относительно базлайна. ~15 падений — пред-существующие sqlite-артефакты (case-insensitive collation, `/`→404 без SSR-индекса, `supplier_id` NOT NULL на sqlite); в идеале со временем довести до нуля. Также: `php -l`, `php artisan route:list`.
- **Frontend (Vue 3/Vite):** `cd frontend && npx vite build && npx vue-tsc --noEmit`. eslint содержит много пред-существующего prettier-шума (не баги) — по возможности форматируй только новые/тронутые файлы.
- `backend/bootstrap/cache/*.php` — генерируемый кэш, не коммить (`git checkout -- backend/bootstrap/cache/`).
- `CLAUDE.md` и `docs/**` **отслеживаются** (исправлено в `.gitignore`); прочие случайные `*.md` в корне игнорируются намеренно. Доки/правила коммить как обычные файлы — `git add -f` больше не нужен.

## 4. Архитектура (кратко)

- `backend/` — Laravel: REST API (`routes/api.php`), Blade-админка и кабинет поставщика (`routes/web.php`), SEO/SSR (`Seo\SpaController`), платежи Cryptomus/Monobank, Telegram-бот, KASM-запуск аккаунтов.
- `frontend/` — Vue 3 + Vuetify + Pinia + Vue Router + i18n (Vite).
- Карта функционала: `docs/functionality/`. Аудит багов и план: `docs/bugs/`, `docs/REMEDIATION.md`.

## 5. Безопасность

- Санитизируй любой серверный/пользовательский HTML перед рендером (фронт: `v-safe-html`/DOMPurify; admin-blade: `escapeHtml`; SSR JSON-LD: `JSON_HEX_TAG`).
- Финансовые операции и вебхуки — идемпотентны (атомарный compare-and-set по статусу транзакции) и под блокировками.
- Проверяй ownership и роли на **каждом** мутирующем эндпоинте.
- Секреты не логируй, не коммить, не отдавай в открытом виде не-главному админу.
